Treetables are for organizing hierarchical data with multiple columns.

Use When

Use this component when you need to display a hierarchical data structure with multiple columns per entry. Make sure all entries use the same columns. E.g. the parent has a name and a description column, the child cannot have a name and a date column.

Do not use

Avoid using this component in any of the following cases:

  • You require using a pagination. The treetable does not support paging.
  • The entries in your datastructure do not have multiple values (columns). Consider using a TreeView.
  • Your elements need to be sorted. Since the data-rows are indented, a change of the order would cause confusion.

Behavior

Each entry can have none, one or multiple descendants. The first column of a descendant-row is indented to visualize the hierarchy. Parent rows are collapsed, i.e. all children rows are hidden initially. To display the children the user can either click on the leading caret-icon or the whole parent row, depending on the row-selection mode. Avoid the row-selection mode when the row contains individually clickable items.

Sorting

You can enable custom sorting by using recursive rendering. A treetable can have multiple root rows (top level). Sorting only occurs within the same level under the same parent: root rows are sorted among themselves and siblings are only sorted against each other.

Filtering

You can enable filtering by using recursive rendering. A filter or multiple filters will be applied to all nodes. The lowest level node clearing the filters is relevant in this case, because it and all its' ancestors will be displayed in the treetable. Descendents will be cut off.

Technical limitations

Note that up to 6 levels of nesting are supported.

Recursive smart iterator

If you want to let the treetable handle data processing needed by its various features, you will need to use the *clrTtItems structural directive instead of creating a template and looping over it. This directive makes it possible to recursively render trees and enables features like sorting, filtering & selection for the rendered dataset.
It has the same syntax as *ngFor and accepts an additional getChildren parameter that receives a node and should return its children. Like *ngFors index exported value, *clrTtItems also provides exported values.
The following exported values can be aliased to local variables:

  • $implicit: T: The value of the individual items in the iterable (clrTtItemsOf)
  • clrTtNode: The value of the model of the row. This needs to be passed to the [clrTtItem] input of the treetable-row, to further render the child rows.
  • isLeaf: True if the current node is a leaf and has no children.

Selection

To allow actions on multiple items at once, the treetable provides the ability for the user to select rows. To make rows selectable in your treetable, you need to use *clrTtItems and do the following:

  • Add a [clrTtItem] input on each <clr-tt-row> component
  • Add a [(clrTtSelected)] two-way binding on the treetable itself, to have access to the list of currently selected items. Note that by adding items to this list yourself, you can dynamically select elements if you need to.

This will enable the default Clarity selection model used in the TreeView. Parent rows will be automatically selected, if all of its child rows are selected.

If needed this selection behaviour can be changed, to no longer automatically select parent rows, if all the child rows are selected. To achieve this the [clrTtAutoParentSelection] input of the treetable must be set to false. Now the parent row will stay in status INDETERMINATE when all of its children are selected as long as the parent was not manually selected.

Filtering

This can be done by providing either a built-in filter or a custom filter, which implements ClrTreetableFilterInterface.
Currently there is only one built-in filter component, a string filter. This component requires that a ClrTreetableStringFilterFunction is passed to it. This is needed to determine which field on the original object has to be filtered for.

A custom filter can be created by implementing the ClrTreetableFilterInterface. The custom implementation then has to be passed as an [clrTtFilter] for a treetable-filter element to add it to a treetable-column. The filter will be applied to the dataset. If you want to make this filter server driven, then you have to return true in the accept() method in the interface and react to the filter state changes in the (clrTtRefresh) output of the treetable.
