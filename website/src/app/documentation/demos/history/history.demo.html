The History is used to navigate back to the recently viewed pages similar to the browser's history. It must be pointed out that the history uses a historical path, rather than Breadcrumb hierarchical one.

Usage

Use History in your application when the last recently visited pages are to high importance for your user.

Further the information structure of your application plays an important part. If the information structure forms a semantic net - the History is recommended if it forms a hierarchical tree, breadcrumbs might be suitable to aid your users navigating.

Features

Pin - The history provides a pin-feature. If the history is pinned it is shown above the page title as a list of links in the history-bar. If the history is unpinned the last visited pages are shown as a dropdown beneath the history icon.

If the users pins the history this setting must be stored to the users-settings/browser-storage/cookie. Default behavior, pinned or unpinned, should be decided on the applications target

Placement

The history icon is placed in the command-bar in the upper right of the page layout.

Behavior

Some basic rules apply to the History

  1. The current page is not displayed
  2. Cycles are not logged, so the same page does not appear multiple times in the path.
  3. Maximum of 3 items are displayed in the path
  4. Browser back (via Browser-functionality or via Back-Link) calls the last recorded page in the history.

Notes

Creation pages (when on the last visited page a new entity is created) should be omitted in the historical path at this either requires to update an already created object or triggers a possibly unwanted additional creation (duplicate) of an object.

Obviously the History AND Breadcrumb SHOULD NOT BE used together on a page.
